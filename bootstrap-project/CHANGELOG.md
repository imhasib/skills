# Changelog

All notable changes to the `bootstrap-project` skill.

## [unreleased] ‚Äî 2026-06-23

### Changed (CI ‚Äî `main` push now always builds + publishes `:latest`, never deploys; `dev` push remains the only auto-deploy trigger at bootstrap)

- **`templates/.github/workflows/build.yml.tmpl`**: dropped the `{{#PROD_READY}}` gate around `main` in the `push.branches` list ‚Äî `main` is now in the build-trigger set from bootstrap onward, not just post-`/configure-prod`. The tag-computation step already produced `sha-<sha>` + moving `latest` on `main` and `sha-<sha>` + moving `dev` on `dev`, so the only behavioural change is *that main builds happen at all*. The `trigger-deploy` job's `if:` stays `github.ref_name == 'dev'` so `main` pushes never roll a staging deploy ‚Äî `:latest` lives in the registry, deploys are deliberate. Feature branches still don't trigger build (only `dev` and `main` push do); CI on feature branches happens via PR.
- **`modules/cicd.md`**: rewrote the `build.yml` triggers section to spell out the per-branch contract (dev ‚Üí build+deploy, main ‚Üí build only, others ‚Üí nothing), and tightened step 5 to note the explicit `if:` guard that keeps `main` builds from dispatching `deploy.yml`.
- **`SKILL.md`** (Phase 4 ‚Üí CI bullet): replaced "main/tag triggers + release.yml are NOT stamped at bootstrap" with the new contract ‚Äî main triggers ship at bootstrap (build only, no deploy), tag triggers + release.yml stay deferred to `/configure-prod`.
- **`templates/deployment/README.md.tmpl`** and **`templates/deployment/CLAUDE.md.tmpl`**: env-table "Auto-deploy" row changed from "GHA on `main` push (when prod_ready)" to "GHA on `v*` tag push (when prod_ready). `main` push only publishes `:latest` ‚Äî no deploy." Aligns the operator-facing table with the actual trigger graph: prod deploys are tag-driven via `release.yml`, never a side effect of merging into `main`.

  Why: until this change, anything merged to `main` at bootstrap produced no image at all (`{{#PROD_READY}}` gated the trigger). That meant `main` was effectively dead state until `/configure-prod` ran ‚Äî no way to consume a stable `:latest` for ad-hoc rollouts, local pulls, or third-party integrations off the canonical branch. Now `main` always produces `:latest` from day one, decoupled from the staging deploy loop. The split between "build is automatic on both branches" and "deploy is automatic on `dev` only" matches the request and keeps prod deploys explicit (tag ‚Üí release.yml ‚Üí deploy.yml).

## [unreleased] ‚Äî 2026-06-23

### Changed (web templates ‚Äî replace NextAuth Google provider with Google Identity Services so the OAuth client secret stops being duplicated into every web app)

- **Move both web templates from NextAuth's `Google` provider to Auth.js v5 `Credentials` provider, with a browser-side `<GoogleSignInButton>` that talks to Google Identity Services (GIS) directly.** The previous design stamped `AUTH_GOOGLE_SECRET` into `templates/web/.env.example.tmpl` and `templates/web-admin/.env.example.tmpl`, plus the operator-facing `dev/env/web-*.env` shape in `modules/deployment.md`. That meant every project bootstrapped from this skill carried the OAuth client secret in **three** places (web-app, web-admin, user-service) instead of one ‚Äî three rotation surfaces, three leak vectors, three sets of CI secret stores to update. GIS lets the browser obtain an `id_token` directly using only the public client ID, which the web app then POSTs to user-service's existing `/auth/google` endpoint (no contract change). The secret now lives only in user-service. Affected files:

  - **NEW** `templates/web/src/components/google-sign-in-button.tsx` ‚Äî reusable `<GoogleSignInButton>`. Loads `https://accounts.google.com/gsi/client` via `next/script`, calls `google.accounts.id.initialize({ client_id, callback })` once the script is ready, and on click calls `google.accounts.id.prompt()` which opens Google's account chooser / One Tap UI. The credential (id_token) flows through an `onIdToken` callback prop, so the component carries no auth logic itself ‚Äî it just produces a token. `prompt()`'s notification listener surfaces `isNotDisplayed` / `isSkippedMoment` reasons via an `onError` prop so a silently-suppressed One Tap (cooldown, FedCM gate) doesn't leave the UI looking inert. Default styling matches the prior plain-button look; an optional `className` prop lets the admin pass theme-token classes.
  - **NEW** `templates/web-admin/src/components/google-sign-in-button.tsx` ‚Äî byte-identical copy of the web component. Could be deduplicated into a shared package later; for now both templates are independent npm projects so a copy keeps the skill stamping behaviour simple.
  - `templates/web/src/auth.ts.tmpl` ‚Äî **rewritten.** Replaces `import Google from 'next-auth/providers/google'` with `import Credentials from 'next-auth/providers/credentials'`. The new provider has `id: 'google'` (so `signIn('google', { idToken })` keeps working from the login form), `credentials: { idToken }`, and an `authorize()` that runs the exact same `exchangeGoogleIdToken()` POST to user-service that the prior `signIn` callback did ‚Äî returning the backend user + token shape Auth.js needs to populate the JWT. The prior `signIn` callback (which returned redirect-URL strings as error codes) is gone; failures log server-side and Auth.js surfaces a generic `CredentialsSignin` to the client. The `jwt` and `session` callbacks keep the same `backendToken` / `backendUser` shape so `src/types/next-auth.d.ts` and every consumer of `session.backendToken` (server-api wrapper, dashboard layout) stay untouched.
  - `templates/web-admin/src/auth.ts.tmpl` ‚Äî same rewrite. Comment header still says "admin console".
  - `templates/web/src/app/login/login-form.tsx` ‚Äî **rewritten.** Drops the in-file Google button + `signIn('google', { callbackUrl })` redirect flow in favour of `<GoogleSignInButton onIdToken={handleIdToken} />`. `handleIdToken` calls `signIn('google', { idToken, redirect: false })` against the Credentials provider, inspects the result, and on success uses `window.location.href = nextPath ?? '/dashboard'` for a full-page navigation (guarantees server components re-render with the fresh auth cookie). Error code map shrinks to `CredentialsSignin` + `Configuration` + passthrough; the OAuth-specific codes (`OAuthSignin`, `OAuthCallback`, `OAuthAccountNotLinked`, `AccessDenied`, `GoogleNoIdToken`) are no longer reachable. Adds a `googleClientId: string` required prop.
  - `templates/web-admin/src/app/login/login-form.tsx` ‚Äî same rewrite. Theme-token button classes (`border-border-muted bg-surface text-text-rich hover:bg-surface-warm`) are passed to `<GoogleSignInButton>` via the new `className` prop so the admin keeps its distinct visual.
  - `templates/web/src/app/login/page.tsx.tmpl` ‚Äî reads `process.env.AUTH_GOOGLE_ID` server-side and passes it down as `googleClientId={...}`. Keeps the env var server-only (no `NEXT_PUBLIC_` prefix needed); the client just receives the string as a prop.
  - `templates/web-admin/src/app/login/page.tsx.tmpl` ‚Äî same.
  - `templates/web/.env.example.tmpl` ‚Äî **`AUTH_GOOGLE_SECRET` removed.** Replaced the two-line `# Google OAuth client credentials (NextAuth)` comment with a four-line block explaining why the client ID is public-by-design and why the secret lives only in user-service. `AUTH_GOOGLE_ID` stays.
  - `templates/web-admin/.env.example.tmpl` ‚Äî same.
  - `templates/web-admin/src/app/login/login-form.test.tsx` ‚Äî **rewritten.** Adds a `vi.mock('@/components/google-sign-in-button')` that returns a static `<button data-client-id={clientId}>` so the test doesn't need to load the GIS script or stub `window.google` inside jsdom. Renders now pass the required `googleClientId` prop. Old `AccessDenied` assertion replaced with `CredentialsSignin` (the new generic failure code).
  - `templates/web/vitest.config.ts` and `templates/web-admin/vitest.config.ts` ‚Äî added `esbuild: { jsx: 'automatic' }`. Without this, vitest's default esbuild config falls back to the classic JSX runtime (which requires `import React` in every `.tsx` test file). Next.js's own SWC build already uses the automatic runtime, so source files compile fine in `next build` / `next dev` without React in scope ‚Äî but vitest didn't get the same treatment. Adding this one line aligns the test runner with the framework and lets the new mock factory in `login-form.test.tsx` use plain JSX. Caught when the rewritten test file failed with `ReferenceError: React is not defined`; turns out the same failure would have hit the *prior* test file too the moment anyone tried to add a second `.tsx` test, because no `.tsx` test had ever actually run in CI with this config (`passWithNoTests: true` masked the absence).
  - `SKILL.md` ‚Äî two lines (Phase 3 web bullets at ~L212 and ~L225). The web bullet now describes the Credentials provider, the new `google-sign-in-button.tsx` component, the absence of `AUTH_GOOGLE_SECRET` from the env shape, and the new `esbuild.jsx: 'automatic'` line in `vitest.config.ts`. The web-admin bullet is similarly updated with a note that the env shape mirrors web (no client secret).
  - `modules/deployment.md` ‚Äî the dev `web-*.env` example block stamped under `dev/env/` no longer includes `AUTH_GOOGLE_SECRET`. The comment above `AUTH_GOOGLE_ID` switches from "Authorized redirect URI to register" to "Authorized JavaScript origin to register" (no path, no trailing slash) and updates the example URLs to the bare `http://localhost:3080` / `http://localhost:3081` forms that GIS requires. Adds a closing paragraph stating the OAuth client secret lives only in user-service.

  Why: the toeic project, which was bootstrapped from this skill, ran into exactly this problem in production-prep ‚Äî three repos all carrying `AUTH_GOOGLE_SECRET` with no clear owner, with `sirr-web-admin` (a different project, hand-built with `@react-oauth/google`) demonstrating that you can get a backend-issued JWT without ever putting the OAuth secret on the frontend. The fix that worked in toeic ‚Äî Credentials provider + browser-side GIS ‚Äî is now the bootstrap default so every new project starts secret-free at the web layer. user-service's contract is unchanged: it still receives `POST /auth/google { idToken }` and returns `{ user, tokens }`. The only thing that moved is *which* process inside the network is the one running the OAuth dance with Google.

  Operator-visible follow-up: the Google Cloud Console OAuth client must have **Authorized JavaScript origins** (not just redirect URIs) listed for each origin that hosts the web app. Existing projects can delete the legacy `‚Ä¶/api/auth/callback/google` redirect URI entries ‚Äî nothing hits them anymore. `modules/deployment.md` calls this out next to the dev env-file shape so it lands in front of an operator before they try to sign in for the first time.

### Added (Q&A-level ‚Äî host-port collision avoidance for parallel dev stacks)

- **Ask for the dev nginx gateway host port at bootstrap; substitute everywhere it surfaces.** Previously the dev `nginx` service hardcoded `"3030:80"` in `templates/deployment/dev/docker-compose.yml.tmpl`. Running two bootstrapped projects on the same dev machine ‚Äî common once a developer has a few karigor-pattern repos ‚Äî failed with `bind: address already in use` on the second `docker compose up`, and the only fix was hand-editing the tracked compose file (untracking the chosen port from the rest of the dev stamps that assume `localhost:3030`). New Q31 in Batch F asks for `Dev nginx gateway host port` with `[default: 3030]`, captured into `BOOTSTRAP.json` as `dev_nginx_port` and propagated via a new `{{DEV_NGINX_PORT}}` placeholder. Affected files:
  - `SKILL.md` ‚Äî adds Q31 to Batch F with rationale; adds `{{DEV_NGINX_PORT}}` to the Phase 4 placeholder list with a `BOOTSTRAP.json` note.
  - `templates/deployment/dev/docker-compose.yml.tmpl` ‚Äî `"{{DEV_NGINX_PORT}}:80"` replaces the hardcoded `"3030:80"` on the nginx service.
  - `templates/deployment/dev/env.example.tmpl` ‚Äî header comment, `APP_BASE_URL`, and `GOOGLE_CALLBACK_URL` all switch to `http://localhost:{{DEV_NGINX_PORT}}` so the env defaults stay consistent with the published port (the Google OAuth callback in particular is registered against the literal URL in the Cloud Console ‚Äî mismatching `APP_BASE_URL` against the actual nginx port silently breaks the OAuth round-trip with `redirect_uri_mismatch`).
  - `templates/deployment/dev/nginx/nginx.conf.tmpl` ‚Äî the dev CORS allow-list switches its self-referential `"http://localhost:3030"` entry to `"http://localhost:{{DEV_NGINX_PORT}}"`. Without this, a same-origin XHR from the dev web app (served by nginx) would get blocked at the preflight because nginx's allow-origin map wouldn't recognise its own published port.
  - `templates/deployment/README.md.tmpl`, `templates/deployment/CLAUDE.md.tmpl`, `templates/core/README.md.tmpl` ‚Äî quick-start URLs and the environments table's "External URL" / "Ingress" rows track the chosen port so a reader landing on either doc sees the actual URL their stack serves on, not a stale `:3030` from a default they overrode.
  - `templates/.claude/agents/deployment-specialist.md` ‚Äî the "Dev binds directly to `3030:80`" explanation line annotates with `{{DEV_NGINX_PORT}}` so the agent stamped into `.claude/agents/` knows the chosen port is the source of truth, not the default.

  Why: collision avoidance is the load-bearing reason ‚Äî the second-project boot failure is the kind of papercut that's invisible until it happens and then takes 10 minutes of grepping through compose to fix. Staging and prod nginx loopback bindings (`127.0.0.1:3030`) deliberately stay at `3030`: they're behind a host-level Caddy reverse proxy that's already pointed at the known loopback port (`reverse_proxy 127.0.0.1:3030` in `staging/REBUILD.md.tmpl`), and prod additionally honours `${NGINX_PORT:-3030}` at deploy time. Touching those would force `/configure-staging` and `/configure-prod` to learn about another variable for no real-world benefit. The dev port was the only port where two parallel bootstrapped projects could actually collide on the same machine, and that's the only port the Q&A now asks about.

### Added (template-level ‚Äî surfaced from real-world bootstrap of `meshpalok`)

- **Stamp `<project>-app/android/app/src/main/AndroidManifest.xml` as an overlay (new template file).** Flutter's default scaffold puts the `INTERNET` permission only in `android/app/src/debug/AndroidManifest.xml` (so `flutter run` works) and never sets `android:usesCleartextTraffic`. The bootstrap default `API_BASE_URL` is plain HTTP at a LAN IP for dev (`http://10.0.2.2:3030/api` on emulator, the host's LAN IP on a physical device) ‚Äî exactly the shape that Android 9+ (API 28+) blocks by default with no log, no error code, and no exception until Dio's `receiveTimeout` fires 15s later. meshpalok hit this on the first physical-device Google sign-in attempt: the `LogInterceptor` printed `*** Request *** uri: http://192.168.0.107:3030/api/auth/google` and then 15 seconds of nothing. The user-visible failure mode is the worst kind ‚Äî "tap the button, nothing happens" ‚Äî because the cleartext block happens at the OS network layer, before any plugin can report it. The new template stamps a complete manifest with `<uses-permission android:name="android.permission.INTERNET"/>` lifted into the main manifest **and** `android:usesCleartextTraffic="true"` on `<application>`, both annotated with the failure mode so the next reader doesn't strip them once HTTPS is in place. Affected files:
  - `templates/app/android/app/src/main/AndroidManifest.xml.tmpl` (new) ‚Äî mirrors Flutter's default manifest with the two additions; uses `{{PROJECT_SNAKE}}` for `android:label` so the orchestrator-substituted value matches Flutter's default labeling convention.
  - `SKILL.md` (the `<project>-app/` bullet under "Code repos") ‚Äî adds "**The overlay must also replace `android/app/src/main/AndroidManifest.xml` (not skip it)**" alongside the existing `.gitignore` overlay instruction, with a one-line rationale citing the silent-hang failure mode. Also adds a new line under "Stamps:" calling out `android/app/src/main/AndroidManifest.xml` as a replacement of Flutter's default.

  Why: the symptom is silent (no exception, no log, no error code) and the diagnostic path is non-obvious (the request shows in Dio's `LogInterceptor` because logging happens client-side before the bytes hit the wire, so it *looks* like the request went out ‚Äî but the OS network stack drops it). meshpalok burned ~30 minutes of debugging on the first sign-in because the request *appeared* successful in logs until the timeout. `android:usesCleartextTraffic="true"` is the right shape for a dev build that uses a LAN backend; the production build will eventually move to HTTPS and can either keep this flag (cleartext won't be used) or narrow with a `network_security_config.xml` allow-list. Downstream projects bootstrapped before this entry will hit the same first-sign-in hang on physical devices ‚Äî the migration is the same two-line edit to their main manifest (lift `INTERNET` out of `debug/AndroidManifest.xml`, add `usesCleartextTraffic="true"`).

- **Wire a `kDebugMode`-gated Dio `LogInterceptor` into `<project>-app/lib/core/network/api_client.dart` at bootstrap.** The template shipped with only `AuthInterceptor` and `_ApiErrorInterceptor` attached, so the first time the bootstrapped app talks to the backend there is *no* visibility into request/response shape ‚Äî the developer sees only the rendered UI state and has to manually add a logger to debug anything that isn't a clean 2xx. meshpalok needed this immediately during the Google sign-in bring-up (to disambiguate "Google SDK threw before we ever called the server" from "we POSTed but never got a response"), and adding it after the fact required editing both the import block and the interceptor list. Stamping a debug-only interceptor at bootstrap saves every downstream project the same 5-minute round-trip. `kDebugMode` gates it cleanly: release builds get no logging (no body leak, no perf cost), debug builds get full `request/requestBody/responseBody/error` lines routed through `debugPrint` (so they participate in Flutter's stdout throttling). Affected files:
  - `templates/app/lib/core/network/api_client.dart.tmpl` ‚Äî adds `import 'package:flutter/foundation.dart'` for `kDebugMode` + `debugPrint`; adds `if (kDebugMode) dio.interceptors.add(_devLogInterceptor());` after the existing interceptors; adds a private `_devLogInterceptor()` factory returning a `LogInterceptor` configured with `requestBody: true`, `responseHeader: false`, `responseBody: true`, `logPrint: (obj) => debugPrint(obj.toString())`. The non-default args avoid `avoid_redundant_argument_values` lint failures on `very_good_analysis`. A block comment above the `if (kDebugMode)` line cites the bring-up-visibility motivation so a future maintainer doesn't strip it as "debug noise".
  - `SKILL.md` (the mobile-app `lib/core/network/` bullet under "Code repos") ‚Äî annotates `api_client.dart` as carrying "a `kDebugMode`-gated `LogInterceptor` so the first end-to-end bring-up isn't blind" so a reader skimming the stamps list knows it's intentional, not a stray dev edit.

  Why: every project bootstrapped from this skill will hit the same "the first HTTP call doesn't work, why?" moment on its first day. Without the interceptor, the answer requires manual logging-shim insertion; with it, the answer is one tap and a log scroll. The cost is zero in release (gated by `kDebugMode`, and Dart's tree-shaker drops `_devLogInterceptor` entirely from release bundles since the call site is statically dead) and ~5 lines of debug-only setup. Downstream projects bootstrapped before this entry can add the same import + 4-line block to their `api_client.dart`; same effect.

### Changed (template-level ‚Äî surfaced from real-world bootstrap of `meshpalok`)

- **Publish dev `mongo` on `127.0.0.1:27017` so host tools (Compass, mongosh) can connect.** The previous template gave the mongo service no `ports:` mapping, so the container was only reachable from inside the `{{PROJECT}}-dev-network` bridge. When a bootstrapper tried to point MongoDB Compass at it from the Windows/macOS host to inspect data or hand-edit a document, the dial timed out with no obvious cause ‚Äî they'd fall back to `docker exec mongosh` and lose the GUI workflow, or worse, edit the tracked compose file in a way that trips the "Never expose internal service ports to the host" hard rule in `templates/deployment/CLAUDE.md.tmpl`. That hard rule was written with staging/prod in mind (where loopback-only Caddy ‚Üí nginx is load-bearing for the TLS topology), but its blanket form swept dev along with it. Affected templates:
  - `templates/deployment/dev/docker-compose.yml.tmpl` ‚Äî adds `ports: ["127.0.0.1:27017:27017"]` to the `{{#MONGO}}` block, with a comment pointing the reader at the `?directConnection=true` URI gotcha (the rs is initiated with `host=mongo:27017`, so without `directConnection` the driver chases that hostname on the host after the initial handshake and stalls topology discovery ‚Äî looks indistinguishable from a network reachability bug).

  Why: meshpalok's owner wanted to use Compass to inspect the `firms` collection after a manual firm-creation sequence, found the mongo container unreachable, and asked whether it was rs-related. It is, partially ‚Äî the rs hostname trap is the *secondary* gotcha once you publish the port; the primary cause was just "no port mapping." A naive fix (`ports: ["27017:27017"]`) would publish on `0.0.0.0` and put dev mongo on the bootstrapper's LAN; the loopback binding keeps it on the developer's own machine, which matches the audience for dev (one developer, one laptop) and preserves the spirit of the deployment CLAUDE.md hard rule. Postgres / MySQL / Redis blocks deliberately left alone for now ‚Äî same pattern would apply if a downstream project asks, but blanket-applying it risks surprising bootstrappers who don't expect their dev redis on a published port. Downstream projects bootstrapped before this entry can copy the one-line `ports:` addition into their own `dev/docker-compose.yml` ‚Äî no rebuild needed beyond `docker compose up -d mongo` to recreate the container.

- **Force the `.gitignore` overlay on `<project>-app/` to actually replace Flutter's default, and broaden the gitignore patterns.** `templates/app/.gitignore` shipped with `*.g.dart` and `*.freezed.dart` (correct), but the orchestrator's overlay step silently kept `flutter create`'s default `.gitignore` instead of replacing it ‚Äî so the bootstrapped meshpalok-app ended up with Flutter's verbose default (sections like `# Miscellaneous`, `# IntelliJ related`, `# Flutter/Dart/Pub related`, `# Symbolication related`) and no `*.g.dart` exclusion. Result: `dart run build_runner build` regenerates `env.g.dart` with whatever's in the developer's local `.env` (their LAN IP for `API_BASE_URL`, etc.), `git add .` picks it up, and the next commit leaks a developer-personal value into shared history. Caught when configuring Google sign-in for meshpalok: the user regenerated `env.g.dart` with `API_BASE_URL=http://192.168.0.107:3030/api` and `git status` showed the change as a normal tracked-file diff, not as an ignored regeneration. Affected files:
  - `templates/app/.gitignore` ‚Äî drops the redundant `lib/core/config/env.g.dart` line (already covered by `*.g.dart`); adds `*.mocks.dart` and `*.config.dart` so future `mockito` / `package_config` generation is covered without another revision; relabels the comment so it matches the meshpalok-app version (`build_runner: envied, riverpod_generator, freezed, json_serializable`).
  - `templates/app/README.md.tmpl` ‚Äî adds a paragraph under "First-time setup" stating that `build_runner` is **required** on every fresh clone (since the generated files are now gitignored, the app will not compile until they're built). Without this, new contributors clone, run `flutter run`, and hit unhelpful "URI doesn't exist" errors against `part 'env.g.dart'` etc.
  - `SKILL.md` (the `<project>-app/` bullet under "Code repos") ‚Äî promotes the `.gitignore` overlay from "Stamps: ... .gitignore ..." to an explicit "**The overlay must replace `.gitignore` (not skip it)**" instruction, with a one-line rationale citing the leak path (Flutter's default omits `*.g.dart`, so personal regenerations end up tracked). Also annotates the `.gitignore` entry in the file list with "**replaces Flutter's default ‚Äî see above**" so a future reader skimming the stamps list can't miss it.

  Why: the template was correct, but the orchestration didn't honor it ‚Äî and the failure mode is silent (no error, just a slow leak of developer-local state into shared commits). meshpalok caught it on day-3 of the bootstrap because the user happened to ask whether `env.g.dart` should be tracked; without that prompt the convention would have drifted further (the project already had three `*.g.dart` files tracked at HEAD, including `auth_provider.g.dart` and `me_provider.g.dart`, both regenerated without thinking). Downstream projects bootstrapped before this entry will need the same migration meshpalok ran: append the four patterns to `.gitignore`, `git rm --cached` the existing `*.g.dart` / `*.freezed.dart` / `*.mocks.dart` / `*.config.dart` files (they stay on disk, just become untracked), commit, and add the README note so new contributors know to run `build_runner` first. The compiled values stay correct as long as each dev runs build_runner against their own `.env` ‚Äî which the README now makes mandatory rather than implicit.

- **Gate `{{PROJECT}}-core` on `user-service` healthcheck (was `service_started`) across dev/staging/prod compose templates.** Previously the dev compose template depended on user-service via `condition: service_started` and the staging/prod templates used the short-form `depends_on: [user-service]` ‚Äî both mean "wait until the container has started," not "wait until the app inside it is ready." Result: on a cold start `{{PROJECT}}-core` boots before `user-service`'s Express server is listening, the first JWKS / token-introspection call from core's auth middleware errors with `ECONNREFUSED`, and the core container either crash-loops or comes up in a wedged state until a manual restart. The fix is to give `user-service` an explicit healthcheck against its own `GET /health` endpoint (which the upstream image already exposes), then gate core on `condition: service_healthy`. Affected templates:
  - `templates/deployment/dev/docker-compose.yml.tmpl` ‚Äî adds a `healthcheck:` block to the `user-service` service (wget `--spider http://localhost:3000/health`, 10s interval, 5s timeout, 5 retries, 15s start_period) and flips `{{PROJECT}}-core`'s `depends_on.user-service.condition` from `service_started` to `service_healthy`.
  - `templates/deployment/staging/docker-compose.yml.tmpl` ‚Äî same healthcheck block on `user-service`; converts `{{PROJECT}}-core`'s short-form `depends_on: [user-service]` to the long-form `user-service: { condition: service_healthy }`.
  - `templates/deployment/prod/docker-compose.yml.tmpl` ‚Äî same change as staging.

  Why: while reviewing meshpalok's `dev/docker-compose.yml` for correct dependency ordering, the user-service block was depending on redis (which it doesn't use ‚Äî its env file has no `REDIS_URL`) and `{{PROJECT}}-core` had no dependency on user-service at all in the local edit, while the template had it gated only on `service_started`. The auth middleware in `{{PROJECT}}-core/src/middleware/jwt-verify.ts` makes a request to user-service on every protected route, so a started-but-not-listening user-service container produces a confusing class of "the API is up but every request 502s" failures on first boot, exactly the regression the longer `start_period` on the new healthcheck is sized for. `wget --spider` was chosen over `curl` and over a node-based probe because alpine-based node images consistently ship `wget` and not `curl` by default, and the spider form returns non-zero on any non-2xx response with no body to parse. The 15s `start_period` matches the observed Express + JWKS warmup window with a small margin; if the upstream user-service image gets slower to boot, this is the knob to bump. Downstream projects bootstrapped before this entry will have the same `service_started` gating in their compose files ‚Äî the migration is the same three diffs applied to their `dev/`, `staging/`, `prod/` compose; no schema migration, no env change.

- **Make the `trigger-deploy` job in `build.yml.tmpl` soft-fail.** GitHub's `workflow_dispatch` API requires the target workflow to exist on the default branch before it'll register the workflow for dispatch ‚Äî and a freshly-bootstrapped project pushes `deploy.yml` only to `dev`, with `main` empty until the first PR lands. So the very first `dev` push hits the build step (succeeds ‚Äî image pushed to DockerHub), then immediately fails at the dispatch step with `422 Cannot trigger a 'workflow_dispatch' on a disabled workflow`, marking the whole run red even though the build itself was fine. The staging deploy is also separately a no-op until `/configure-staging` populates the `<FILL_IN_*>` placeholders. Wrapping the dispatch in `continue-on-error: true` plus a `gh ... || warning` shell pattern lets the build's overall status reflect the build+push outcome (which is what bootstrappers actually care about on day 1), while still emitting a visible warning that surfaces the dispatch state on the run page. Affected templates:
  - `templates/.github/workflows/build.yml.tmpl` ‚Äî `trigger-deploy` job and its inner `Dispatch deploy` step both get `continue-on-error: true`. The `gh workflow run` call is wrapped in `if ! ... ; then echo ::warning:: ...; fi` so a 404/422/403 from the API surfaces as a CI warning instead of a step-failure exit code.

  Why: meshpalok's first end-to-end-green build (after the husky/replSet/permissions/Decimal128 chain of fixes) finally pushed the docker image to DockerHub successfully, only to fail-the-run at the dispatch step with the 422. The bootstrapper had no way to make that step succeed without first running `/configure-staging` AND merging dev ‚Üí main, which is wrong shape: you want the build to be green from commit 1, and the deploy dispatch to be a "nice to have" that warns when it can't fire. Downstream projects bootstrapped before this entry will hit the same first-build-failed-at-dispatch experience ‚Äî the migration is the same diff applied to their `build.yml`, or they can flip the workflow permissions setting plus merge to main (same end result, more steps).

- **Stop stripping trailing zeros in `toDecimal()` ‚Äî preserve input precision through to Decimal128.** The template's `templates/core/src/lib/decimal.ts` validates input via `new Decimal(value)`, then re-emits the string as `d.toFixed()` before handing it to `Decimal128.fromString(...)`. decimal.js's `toFixed()` with no argument normalises the value to its minimum string form ‚Äî which strips trailing zeros. So `toDecimal('10000.00')` produces `Decimal128('10000')` (not `'10000.00'`), and the API round-trip returns `{ $numberDecimal: '10000' }` instead of `{ $numberDecimal: '10000.00' }`. For a money app this is the wrong default: Decimal128 itself preserves the precision of its input (a deliberate BSON feature for currency ‚Äî `'125.50'` is distinct from `'125.5'` for display and audit), so the wrapper should pass the original string through after validation, not normalise it. Affected templates:
  - `templates/core/src/lib/decimal.ts` ‚Äî body of `toDecimal()` changed so the post-validation `normalized` string preserves the input form: for `string` inputs, `value.trim()` (no precision loss); for `number` inputs, `d.toString()` (decimal.js's lossless string form, since the input was a JS number and didn't have explicit precision). The decimal.js validation step stays the same ‚Äî it still throws `'Invalid decimal value: ...'` on non-finite/non-numeric input.

  Why: meshpalok's integration tests posted `amount: '125.50'`, `amount: '10000.00'`, `amount: '75.50'` and asserted the response echoed them back; every assertion failed with `expected '125.5' to be '125.50'` (and similar) once the test harness finally ran end-to-end. The bug was masked for weeks because the integration test infrastructure hadn't been working (replSet self-host issue, supertest agent shape) ‚Äî once those were fixed, the real Decimal128 round-trip bug surfaced immediately. The template fix is shaped: every money-tracking project bootstrapped from this skill will use `toDecimal()` on user-supplied amounts, and every one of them wants `'10.00'` to round-trip as `'10.00'`, not `'10'`. Downstream projects bootstrapped before this entry will silently store amounts with stripped trailing zeros ‚Äî the migration is the same 1-line diff applied to their `src/lib/decimal.ts`, plus a one-off backfill for any already-stored values that need re-precision (typically not worth doing unless display/audit specifically needs it).

- **Add a `permissions:` block to `build.yml.tmpl` so the build job can dispatch `deploy.yml`.** The CI/CD template's `build.yml` ends with a "trigger-deploy" job that calls `gh workflow run deploy.yml` (or its octokit equivalent) on every successful build of `dev`. That dispatch needs `actions: write` on the workflow's `GITHUB_TOKEN` ‚Äî without it, the API call comes back `403 Resource not accessible by integration` and the build job is marked failed even though the docker image was built and pushed successfully. The default token permissions on a repo with "Read repository contents and packages permissions" set as the GITHUB_TOKEN default (the post-2023 GitHub default for new repos) lack `actions: write`. The cleanest fix is to declare the minimum required permissions explicitly on the workflow, rather than asking every bootstrapper to flip the repo-level setting. Affected templates:
  - `templates/.github/workflows/build.yml.tmpl` ‚Äî adds a top-level `permissions: { contents: read, actions: write }` block between `on:` and `jobs:`. `contents: read` keeps `actions/checkout` working (writes aren't needed at the workflow scope here); `actions: write` is what `workflow_dispatch` requires.

  Why: meshpalok's build pushed the docker image to DockerHub successfully on the first end-to-end-green run, then immediately failed at the `Dispatch staging deploy` step with the 403. The error message was load-bearing ‚Äî without the explicit permissions block, the failure mode looks like "the dispatch step has a bug" rather than "the token is read-only by default". Setting the permissions in the workflow file makes the intent explicit and survives org-default changes. Downstream projects bootstrapped before this entry can add the same block manually, or flip the repo-level "Workflow permissions" setting to "Read and write"; both work, the workflow-scoped form is more legible.

- **Add `--ignore-scripts` to the production-stage `npm ci` in `Dockerfile.tmpl`.** The runtime stage runs `npm ci --omit=dev` to install only production dependencies, but the bootstrapped `package.json` ships a `"prepare": "husky"` lifecycle script (used to install git hooks locally on developer machines). When `--omit=dev` skips installing husky (it's in `devDependencies`), npm still tries to run the `prepare` script ‚Äî and `sh -c husky` fails with exit code 127 ("husky: not found") because the binary was never installed. The builder stage doesn't hit this because it installs everything including husky as a devDep; husky 9 then silently no-ops inside Docker because there's no `.git` directory, so the builder stage is silently fine. Only the runtime stage explodes. `--ignore-scripts` skips lifecycle scripts on this npm ci ‚Äî safe because the runtime stage has no real build steps and any dependency-shipped postinstall scripts would already have run in the builder stage. Affected templates:
  - `templates/core/Dockerfile.tmpl` ‚Äî line 19: `{{PACKAGE_MANAGER}} ci --omit=dev` ‚Üí `{{PACKAGE_MANAGER}} ci --omit=dev --ignore-scripts`. The builder-stage `npm ci` on line 6 stays as-is so any pre-build native module compilation still runs.

  Why: meshpalok's first successful DockerHub login (after the `vars`-not-`secrets` fix above) immediately hit `npm error code 127` on the runtime-stage `npm ci`. The husky+`--omit=dev` interaction is a well-known footgun, and every project bootstrapped from this skill ships the same `prepare: "husky"` hook, so the template fix is `--ignore-scripts` on the runtime install. Downstream projects bootstrapped before this entry will hit the same failure on their first successful build ‚Äî the migration is the same one-flag diff applied to their Dockerfile.

- **Pull the DockerHub username from `vars`, not `secrets`, in `build.yml.tmpl`.** The template stamped `username: ${{ secrets.DOCKERHUB_USERNAME }}` next to `password: ${{ secrets.DOCKERHUB_TOKEN }}`, which forced the bootstrapper to create the username as a repo Secret. DockerHub usernames are public (they appear in every pushed image tag ‚Äî `imhasib/meshpalok-core:...`) and storing public values as secrets makes them mutually-uninspectable on the repo Settings page (you can't see what you set, can't tell if it's empty, and the action's `with:` printout silently drops the line when the secret is empty ‚Äî which is exactly how meshpalok's first build failed with "Username required" while showing only `password: ***`). Switching the username to `vars.DOCKERHUB_USERNAME` lets the bootstrapper add it as a repo Variable, where it's visible on the settings page, easy to inspect, and the workflow's `with:` block prints `username: imhasib` so a missing/typoed value fails loudly. Affected templates:
  - `templates/.github/workflows/build.yml.tmpl` ‚Äî `${{ secrets.DOCKERHUB_USERNAME }}` ‚Üí `${{ vars.DOCKERHUB_USERNAME }}`. `DOCKERHUB_TOKEN` stays in `secrets` because the access token *is* sensitive.

  Why: meshpalok's first push to dev failed at the DockerHub login step with "Username required" ‚Äî the secret had been added as a repo Variable (correctly, for a public value), but the workflow read it from `secrets`, so the resolution returned empty and the action errored. The fix is template-shaped: every project bootstrapped from this skill will have the same public-username-vs-private-token split, and the template should reflect that by default. Downstream projects bootstrapped before this entry will keep working if they added the username as a Secret; they can migrate by moving the value to a Variable (or leave it ‚Äî both work, the Variable form is just more legible).

- **Drop the `thresholds: { lines: 80, functions: 80, branches: 70, statements: 80 }` gate from `templates/core/vitest.config.ts`.** The template shipped with an aspirational 80% coverage gate that turned every freshly-bootstrapped project's first CI run red ‚Äî newly-bootstrapped projects have a handful of unit tests (the seed scaffold ships ~7 around `serialize.ts`) and one or two real route tests, putting actual coverage near 0.5%. The 80% threshold then fails CI on the very first push, before the project has any real feature code to cover, and forces the bootstrapper to either disable the gate locally or write throwaway tests just to make CI green. Removing the gate keeps the coverage reporter (`text` + `lcov` outputs still print at the end of every `vitest --coverage` run) so teams can watch the trend without CI-blocking on a number that's wrong by definition until the project has critical mass. Affected templates:
  - `templates/core/vitest.config.ts` ‚Äî `thresholds: { ... }` line removed. The `coverage.provider`/`reporter`/`exclude` block stays so `npm test -- --coverage` still emits a report.

  Why: meshpalok's CI failed on day-one with "Coverage for lines (0.48%) does not meet global threshold (80%)" ‚Äî a real number for a real bootstrapped repo with no business logic yet. The threshold is the wrong shape for a *newly-bootstrapped* project: it would be sensible at the 30-route-mark, but at the 7-test-mark it just blocks pushes. A future template addition could reintroduce a gate gated by repo age or test count, but for now the right default is no gate. Downstream projects bootstrapped before this entry can choose: delete the `thresholds:` line manually, or grow tests to meet it.

- **Rename the dev moving image tag from `dev-latest` to `dev`.** The template stamped two coordinated `dev-latest` references ‚Äî the `moving_tag` output in `build.yml.tmpl` and the `${CORE_TAG:-dev-latest}` default in both dev and staging `docker-compose.yml.tmpl` ‚Äî plus the `default: dev-latest` on `deploy.yml.tmpl`'s `image_tag` input and the docs line in `modules/cicd.md`. The `-latest` suffix was redundant with the `dev` branch already implying "latest dev build" and produced noisy tag names in DockerHub UI (`dev-latest`, `latest`, `sha-abc1234` side-by-side). meshpalok renamed all five references in one PR on 2026-06-19; this entry lifts the rename into the template. Affected templates:
  - `templates/.github/workflows/build.yml.tmpl` ‚Äî `moving_tag=dev-latest` ‚Üí `moving_tag=dev` (line ~35). `main`/tag branches still emit `latest` ‚Äî only the dev moving tag changes.
  - `templates/.github/workflows/deploy.yml.tmpl` ‚Äî the `image_tag` workflow_dispatch input default changes from `dev-latest` to `dev` so a manual deploy without an explicit SHA pulls the dev moving tag.
  - `templates/deployment/dev/docker-compose.yml.tmpl` and `templates/deployment/staging/docker-compose.yml.tmpl` ‚Äî `${CORE_TAG:-dev-latest}` ‚Üí `${CORE_TAG:-dev}` so compose falls back to the moving dev tag when `CORE_TAG` isn't pinned by the deploy job.
  - `modules/cicd.md` ‚Äî the "Compute tag" step now reads `(`dev` for dev, `latest` for main)` instead of `(`dev-latest` for dev, ‚Ä¶)` to match.

  Why: the `-latest` suffix was a no-op disambiguator ‚Äî there's no `dev-stable` or `dev-rc` to disambiguate against in this template, and the four coordinated references all needed to move together every time the convention shifted. meshpalok's `meshpalok-core/.github/workflows/build.yml` was the first place this got renamed, and the discovery that all five surfaces had to flip in lockstep is the template-shaped lesson. Downstream projects bootstrapped before this entry will keep working on `dev-latest` until they decide to rename ‚Äî the migration is a four-file find-and-replace plus rebuilding the dev image once to refresh the moving tag.

- **Run dev mongo as a single-node replica set + pin `?replicaSet=rs0&directConnection=true` on the dev `MONGODB_URI`.** The dev compose template shipped mongo as a standalone (`image: mongo:7` with no `command:`), and the dev env template emitted `MONGODB_URI=mongodb://mongo:27017/{{PROJECT}}` with no query params. That worked exactly until the first time `{{PROJECT}}`-core opened a multi-document transaction ‚Äî at which point the MongoDB driver refused with "Transaction numbers are only allowed on a replica set member or mongos" and dev integration tests started red-bar on routes the production stack would actually need to ship (animal sale ‚Üí expense+income atomic write, income create with idempotency lock, etc.). meshpalok hit this on day-2 of the bootstrap and silently patched both the compose file and the URI in the live repo; this entry lifts that patch into the template. Affected templates:
  - `templates/deployment/dev/docker-compose.yml.tmpl` ‚Äî mongo service now runs `command: ["--replSet", "rs0", "--bind_ip_all"]` with a healthcheck that does `try { rs.status() } catch { rs.initiate({...}) }` so the replica set self-initiates on first boot and is idempotent on subsequent boots. `start_period` raised to 20s and `retries` to 10 to give the initiate enough time on a cold container. Block comment cites the multi-document-transaction reason so the next reader doesn't strip `--replSet` thinking it's dead weight.
  - `templates/deployment/dev/env.example.tmpl` ‚Äî `MONGODB_URI` line now stamps `mongodb://mongo:27017/{{PROJECT}}?replicaSet=rs0&directConnection=true` and a comment above it cites the exact driver error you'd get without the params. `directConnection=true` keeps the dev URI working when the host can only resolve `mongo` (the compose-network DNS name) and not the replica set's advertised host ‚Äî without it, the driver does SDAM discovery and tries to reconnect to whatever `mongo:27017` advertised itself as, which on first init can briefly be the container's hash hostname.

  Why: meshpalok's `meshpalok-deployment/dev/docker-compose.yml` and `dev/env.example` were both hand-patched to this shape on 2026-06-18 after the dev env review surfaced the missing `?replicaSet=rs0` on `dev/env/meshpalok-core.env` (which silently inherited the template's standalone-friendly URI). The patch is template-shaped: every `{{PROJECT}}`-core bootstrapped from this skill will eventually open a transaction (the integration scaffold and `meshpalok` PRD #6.3 both assume it), so a standalone-mongo dev stack is wrong by default. Downstream projects bootstrapped before this entry will keep working until their first transactional route lands; the migration is the diff applied to `meshpalok-deployment` on 2026-06-18 (compose `command:` + healthcheck swap and the URI query-param append).

### Changed (template-level ‚Äî surfaced from real-world bootstrap of `meshpalok`, 2026-06-18)
